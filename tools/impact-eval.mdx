`impact-eval` is the scoring service for eval modules.

## Runtime Model

- Subscribes to `trace_insertion` events
- Resolves workspace/version/trace context
- Loads active `IMPACT_EVAL` modules
- Executes scoring asynchronously per module
- Stores results as runs

## Scoring Implementations

### `onetofive`

- Builds/calibrates one-to-five style scoring
- Supports module initialization and alignment helpers
- Produces score, explanation, and component-level details

### `flag`

- Verdict-style scoring path
- Produces explanation and optional flagged traces

## Module Interface Pattern

Implementations follow a shared interface:

- `initialize_module(...)`
- `align_module(...)`
- `run_evaluation(...)`

This keeps module strategy extensible while preserving a stable orchestration path.

## Health and Operation

The public API is intentionally minimal.

- `GET /health` reports subscription state

Most work occurs in background event handlers rather than request/response endpoints.

## How Results Are Stored

Each run is linked to:

- `version_id`
- `module_id`
- `interaction_id`

Run payload includes score/verdict, explanation, and component data used by Analyse/Verify.

## Related Services

- `impact-telemetry` and `impact-metric` provide upstream data
- `impact-lens` runs parallel tagging workflows
- `impact-studio` manages module authoring inputs

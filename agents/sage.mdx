## Working in Impact Studio

Sage is your companion in Impact Studio, guiding module creation:

### Starting a Module

```
You: I want to create a module for detecting hallucinations

Sage: Great choice. Hallucination detection is critical for 
maintaining trust. Let me help you design this module.

**First, let's clarify what "hallucination" means for your use case:**

1. **Factual errors** - Incorrect information
2. **Unsupported claims** - True info not in source documents
3. **Fabricated references** - Made-up citations or quotes
4. **Confident uncertainty** - Stating guesses as facts

Which types are most important for your product?

You: Mainly unsupported claims and confident uncertainty

Sage: Got it. I'll focus the module on detecting:
- Claims not grounded in provided context
- Overly confident statements about uncertain information

This is important for your RAG system where responses 
should be traceable to source documents.
```

### Designing Rubrics

```
Sage: Now let's design the evaluation rubric.

**Recommended: Pass/Fail Evaluation**

For hallucination detection, I recommend Pass/Fail rather 
than a scale because:
- Hallucinations are binary (present or not)
- Easier to act on alerts
- Clearer threshold for quality

**Pass (No Hallucination):**
- All claims are supported by provided context
- Uncertainty is acknowledged appropriately
- No fabricated information

**Fail (Hallucination Detected):**
- Contains unsupported claims
- States uncertain information as fact
- Includes fabricated details

Does this align with your expectations?

You: Can we add a "Borderline" category for edge cases?

Sage: Good idea. I'll add a three-tier system:

**Pass** - Fully grounded, appropriate confidence
**Borderline** - Minor unsupported details, mostly accurate
**Fail** - Clear hallucination or fabrication

This gives you nuance while maintaining actionability.
Borderline cases can trigger review without blocking.
```

### Refining Configuration

```
Sage: Let's configure the module settings.

**Module: Hallucination Detection**

| Setting | Recommended | Reason |
|---------|-------------|--------|
| Importance | High | Critical for trust |
| Alertable | Yes | Catch issues quickly |
| Alert threshold | Any "Fail" | Zero tolerance |
| Starred | Yes | Key metric |

**Evaluation Focus:**
The module will examine:
1. Response content vs. provided context
2. Confidence level of statements
3. Presence of citations or hedging

**Additional Settings:**
- Include component analysis: Yes
- Capture reasoning: Yes (for debugging)

Look good?
```

### Testing Before Deploy

```
You: Can we test this before deploying?

Sage: Absolutely. Let me run it against some samples.

**Test Results (10 samples):**

| Sample | Expected | Result | Match |
|--------|----------|--------|-------|
| #1 | Pass | Pass | ✓ |
| #2 | Fail | Fail | ✓ |
| #3 | Pass | Pass | ✓ |
| #4 | Borderline | Pass | ~ |
| #5 | Fail | Fail | ✓ |
| ... | | | |

**Accuracy: 90%**

Sample #4 disagreement:
- Expected: Borderline (minor unsupported detail)
- Result: Pass

The model is slightly lenient on minor cases. Options:
1. Accept (minor cases less critical)
2. Adjust rubric to be stricter
3. Add examples to clarify borderline cases

What would you prefer?
```

## Common Module Types

Sage has expertise in common evaluation patterns:

### Quality Modules

```
You: Help me create a quality evaluation suite

Sage: I'll help you set up a comprehensive quality suite.
Here are the key modules I recommend:

**Core Quality:**
1. Response Helpfulness - Does it answer the question?
2. Factual Accuracy - Is the information correct?
3. Completeness - Are all aspects addressed?
4. Clarity - Is it easy to understand?

**Communication:**
5. Tone Appropriateness - Professional and empathetic?
6. Conciseness - Appropriately brief?

**Safety:**
7. Harmful Content Detection - Any unsafe content?
8. PII Detection - Personal information exposed?

Which would you like to start with?
```

## Alignment Best Practices

Sage helps ensure modules align with business goals:

### Business Goal Mapping

```
You: Our main goal is reducing support tickets

Sage: Good context. Let me suggest modules aligned with 
ticket reduction:

**Primary Modules:**
1. **Resolution Detection** - Did we solve the issue?
   → Directly measures successful interactions

2. **Self-Service Enablement** - Did we teach the user?
   → Reduces repeat questions

3. **Escalation Detection** - Human handoff needed?
   → Identifies gaps in automation

**Secondary Modules:**
4. **First Contact Resolution** - Solved in one turn?
5. **Knowledge Gap Detection** - Missing information?

These directly connect to your ticket reduction goal.
Want me to create them with that alignment in mind?
```

### Calibration Assistance

```
You: Our human reviewers disagree with the module sometimes

Sage: Let's calibrate. Can you share some examples where 
the module disagreed with human judgment?

You: [shares 5 examples]

Sage: I see the pattern. The module is:
- Stricter on technical accuracy than humans
- More lenient on tone than humans

**Calibration Options:**

1. **Adjust rubric** - Make criteria match human judgment
2. **Add examples** - Show borderline cases
3. **Split module** - Separate technical and general accuracy

I recommend option 3 for your case. Technical queries 
may need different criteria than general questions.

Should I create separate modules?
```

Modules are evaluation units attached to categories/dimensions and executed against interactions.

## Core Module Types

### Impact Eval (`impact_eval`)

Used for scored judgments.

Current scoring implementations in the backend service:

- **`onetofive`**: calibrated one-to-five scoring
- **`flag`**: verdict-style flagging with explanations and optional flagged traces

Runs are stored with score/explanation/component breakdown in `runs.result`.

### Impact Lens (`impact_lens`)

Used for tagging/classification.

The lens runtime tags interactions and traces for active lens modules and upserts one run per interaction/module with incremental updates.

## Module Structure

A module typically includes:

- `name`
- `type`
- `category_id`
- `dimension_id`
- `components[]`
- `metadata` (rubric, prompt, thresholds, and status config)

## Module Lifecycle

From current schema and runtime behavior:

- Schema-level status: `live` or `archived`
- Runtime config often supports enable/disable toggles in metadata
- Only enabled/live modules should be part of active processing

## Where Modules Are Built

Use [Impact Studio](/platform/impact-studio) to create/update modules and persist them through `store_module`.

## Execution Flow

1. New traces/interactions arrive
2. `impact-eval` and/or `impact-lens` select active modules
3. Service runs evaluation/tagging
4. Result is stored as a run
5. Analyse/Verify views read and aggregate results

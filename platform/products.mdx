Products and versions are the control plane for data ingestion and evaluation scope.

## Data Model

```text
Workspace
└── Product
    └── Version
        ├── API Key
        ├── Interactions
        ├── Traces / Spans
        └── Runs
```

## Why Versions Matter

Each version can represent a model/prompt/runtime configuration. This lets you:

- Compare quality before/after changes
- Roll out gradually
- Keep telemetry and evaluation results attributable to specific releases

## API Key Scope

API keys are created against a workspace + version.

When telemetry arrives at `impact-telemetry`, that key resolves workspace and version context used downstream in metric and evaluation pipelines.

## Typical Setup Flow

<Steps>
  <Step title="Create workspace">
    Done during onboarding in the app (or via admin service).
  </Step>
  <Step title="Create product">
    Define the AI product you want to monitor and evaluate.
  </Step>
  <Step title="Create version">
    Add the specific deployable runtime configuration.
  </Step>
  <Step title="Create API key">
    Use the key in your SDK-instrumented app.
  </Step>
</Steps>

## Operational Best Practices

- Keep version names aligned with real releases (`v1.2.0`, `prod-2026-03-01`, etc.).
- Rotate API keys regularly.
- Use separate versions for staging and production.
- Keep module and run analysis version-aware.

## Related Services

- **Admin service**: workspace, membership, invitations, API keys, setup endpoints
- **Telemetry service**: OTLP ingestion bound to workspace/version
- **Metric service**: trace and interaction metrics per version

Impact AI analysis is built from a service pipeline, not a single dashboard query.

## Pipeline Summary

1. **Ingest**: `impact-telemetry` receives OTLP traces/logs
2. **Normalize**: `impact-metric` expands batches into normalized spans
3. **Aggregate**: `impact-metric` builds traces and interaction metrics
4. **Evaluate**: `impact-eval` and `impact-lens` create runs
5. **Present**: Analyse/Verify views read projections and run outputs

## Runtime Services Involved

- `impact-telemetry`: `/v1/traces`, `/v1/logs`
- `impact-metric`: continuous `batch2span` and `spans2trace` processors
- `impact-eval`: event-driven module scoring
- `impact-lens`: event-driven interaction/trace tagging
- `impact-issue`: diagnosis and aggregation endpoints for issue analysis

## What You See in the UI

### Analyse

- **Monitor**: health and trend surfaces
- **Datasets**: interaction/trace exploration
- **Catalog**: module-level view
- **Issues**: grouped and interaction-level diagnostics

### Verify

- Quality/capabilities/outcomes views sourced from run outputs and module metrics

## Metrics and Runs

- Canonical trace/interaction metrics are written by `impact-metric`
- Evaluation results are written as runs (`version_id`, `module_id`, `interaction_id`)
- Frontend components combine both metrics and run data for charts and summaries

## Operational Note

Because evaluation is asynchronous and event-driven, ingestion is near real-time but not strictly synchronous with the originating request.
